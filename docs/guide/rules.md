# 转发规则

转发规则是用户真正访问的入口。

进入：

```text
转发规则 -> 添加规则
```

## 基本配置

| 配置项 | 说明 |
| --- | --- |
| 规则名称 | 用来识别这条规则 |
| 协议 | TCP、UDP 或 TCP+UDP |
| 入口端口 | 用户访问的端口 |
| 目标地址 | 最终服务 IP 或域名 |
| 目标端口 | 最终服务端口 |
| 转发方式 | iptables、nftables、realm、socat、gost 等 |
| 所属用户 | 规则归属和流量计费对象 |

例如：

```text
入口端口：15201
目标地址：1.2.3.4
目标端口：5201
协议：TCP
```

访问：

```text
入口服务器IP:15201
```

就会转发到：

```text
1.2.3.4:5201
```

## 转发模式

| 模式 | 路径 | 适合场景 |
| --- | --- | --- |
| 主机端口转发 | 用户 -> 主机 -> 目标 | 单台主机即可完成转发 |
| 隧道转发 | 用户 -> 入口 -> 隧道 -> 出口 -> 目标 | 入口和出口不同，或需要加密链路 |
| 端口转发链 | 用户 -> 入口 -> 中转 -> 出口 -> 目标 | 固定多跳路径 |
| 转发组 | 多入口 -> 同一目标 | 多入口高可用和 DDNS 故障转移 |

## 转发方式怎么选

| 方式 | 适合场景 |
| --- | --- |
| iptables | 内核转发，性能好，适合常规 IPv4 转发 |
| nftables | 新系统推荐，IPv4/IPv6 管理更统一 |
| realm | 用户态转发，配置简单 |
| socat | 兼容性好，适合简单场景 |
| gost | 功能更多，适合特殊代理或隧道能力 |

如果不确定，优先使用面板推荐或默认方式。

## 规则状态

规则状态常见有：

- 在线：规则已经正常运行。
- 离线：规则未运行或执行失败。
- 检测中：正在同步或探测。
- 停用：规则被手动关闭。

创建规则后建议点击链路测试或自测，确认入口到目标是否连通。

## fast-open 和 zero-copy

部分转发方式支持 TCP Fast Open 或 zero-copy。

- TCP Fast Open：减少 TCP 建连等待，部分场景能降低连接延迟。
- zero-copy：减少数据复制，部分场景能降低 CPU 消耗。

注意：

- 不是所有转发方式都支持。
- 不支持时开关会置灰。
- 这些功能通常只对 TCP 有意义。
- 如果开启后出现兼容性问题，可以关闭再测试。

## UDP over TCP

`UDP over TCP` 用于中转链路屏蔽或限制 UDP 的场景。开启后，用户访问入口端口的 TCP 流量仍走原来的 ForwardX 自定义加密隧道，UDP 流量会额外通过一条伪装 TCP 的通道传到出口，再由出口还原为 UDP 转发到目标服务。

使用条件：

- 只支持“隧道转发”中的 ForwardX 自定义加密隧道。
- 规则协议需要选择 `TCP+UDP`。
- 入口机和出口机的 Agent 都需要升级到支持该功能的版本。
- Agent 需要 root 权限运行，因为伪装 TCP 通道需要使用 raw socket。

端口说明：

- 不开启 `UDP over TCP` 时，规则继续使用原来的单端口隧道。
- 开启后需要一个“UDP over TCP 端口”，这是入口机到出口机之间的专用通道端口，不是用户直接访问的入口端口。
- 端口可以手动填写；留空或填写 `0` 时，面板会在出口机器上自动分配一个可用端口。
- 如果出口机器安全组或防火墙有限制，需要放行这个专用端口的 TCP 流量。

适合场景：

- 中转机或线路不允许直接传输 UDP。
- 游戏、语音等小流量 UDP 业务需要通过受限链路。

如果本来原生 UDP 线路质量较差，该功能不会修复公网自身丢包；它只负责让 UDP 数据通过可用的伪装 TCP 通道到达出口。
